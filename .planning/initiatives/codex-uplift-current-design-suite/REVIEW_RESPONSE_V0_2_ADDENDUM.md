# REVIEW_RESPONSE v0.2 Addendum: Product Framing, Autonomy Profiles, and Setup-Assistant Direction

Date: 2026-04-30

## 1. Framing correction

The earlier response said the review “implicitly changes the product category” from a bootstrap bundle to a setup assistant. That phrasing was too reviewer-centered.

A more accurate framing is:

> The review makes explicit a product category that was already implicit in the user's stated goals. v0.1 was implemented as a conservative bootstrap bundle, but the surrounding ambition was broader: a Codex setup/onboarding system for user-level posture, project-level initialization, skills, agents, hooks, permissions, sandboxing, and autonomy profiles.

Therefore, the review should not be treated as arbitrarily raising the bar. It is evaluating the v0.1 package against the larger product contract implied by the project. The fair distinction is:

- **Product direction:** Codex setup/onboarding assistant.
- **v0.1 implementation slice:** conservative bootstrap bundle.
- **v0.2 obligation:** first credible setup-assistant release.

This changes the response to the review in two ways.

First, product-contract findings become more central. Config doctor, candidate generation, component selection, install modes, plugin path validation, duplicate-skill detection, and autonomy presets are not merely polish. They are the first version of the product that the user was implicitly asking for.

Second, the response should be less defensive about the package being “just v0.1.” v0.1 can still be a reasonable first slice, but the roadmap should explicitly converge on the larger setup-assistant product rather than treating the reviewer’s expectations as external scope expansion.

## 2. Revised response posture

The response to the reviewer should be organized around four distinct categories:

1. **Accepted implementation defects** — issues in the existing package regardless of product ambition.
2. **Accepted setup-assistant requirements** — requirements that become necessary once the user's intended product category is made explicit.
3. **Platform-modeling corrections** — places where the review or earlier surrounding discussion may have misunderstood Codex mechanics.
4. **Open empirical questions** — behaviors that should be probed locally instead of asserted from memory.

This separation matters because not every disagreement has the same status. For example:

- Plugin path validation is an implementation defect.
- Config doctor is a setup-assistant requirement.
- “Hooks are not real Codex hooks” would be a platform-modeling error.
- The exact practical behavior of `danger-full-access + auto_review` under different task types should be locally probed.

## 3. v0.2 product thesis

v0.2 should define `codex-uplift-kit` as:

> A non-destructive Codex setup and posture assistant that installs reusable agentic-development assets, inspects existing Codex configuration, proposes safe/adaptable configuration profiles, initializes project-level Codex surfaces from repo evidence, and supports higher-autonomy operation with explicit compensating controls.

The key word is **posture**. The package should not merely copy files. It should help the user choose and inspect an operating posture: conservative, review-oriented, autonomous, full-access-reviewed, or externally isolated.

## 4. v0.2 non-goals

v0.2 should not:

- silently overwrite existing `~/.codex` or `~/.agents` files;
- silently enable hooks, rules, plugins, network, or full access;
- silently switch users to `approval_policy = "never"`;
- present `danger-full-access + auto_review` as equivalent to sandboxed safety;
- treat project-level config as safe to load before the project is trusted;
- install duplicate standalone and plugin skills without warning;
- claim exact effective Codex behavior where it has not been observed locally.

## 5. v0.2 command surface

Recommended CLI commands:

```text
codex-uplift-init inspect
codex-uplift-init install [--mode classic|plugin|hybrid] [--components ...]
codex-uplift-init config doctor
codex-uplift-init config candidate --profile <profile>
codex-uplift-init project inspect
codex-uplift-init project candidate --profile <profile>
codex-uplift-init rules candidate --profile <profile>
codex-uplift-init hooks candidate --profile <profile>
codex-uplift-init verify
codex-uplift-init uninstall --manifest <path>
```

### `inspect`

Produces a structured inventory of the current user-level setup:

- `~/.codex/AGENTS.md`
- `~/.codex/AGENTS.override.md`
- `~/.codex/config.toml`
- `~/.codex/hooks.json`
- `~/.codex/rules/`
- `~/.codex/agents/`
- `~/.codex/plugins/`
- `~/.agents/skills/`
- `~/.agents/plugins/marketplace.json`
- known duplicates and collisions
- candidate files previously generated by the kit

The output should distinguish:

- installed by this package;
- existing before this package;
- collision;
- candidate pending review;
- unknown/unmanaged.

### `config doctor`

Reads local config and reports what can be observed:

- active base keys such as `approval_policy`, `approvals_reviewer`, `sandbox_mode`, `default_permissions`, `allow_login_shell`, and `profile`;
- defined profiles;
- named permission profiles;
- sandbox workspace-write network access;
- hooks feature flag;
- app/tool approval defaults where present;
- project trust entries;
- managed or requirements constraints if visible;
- warnings where local desired settings may be overridden by managed configuration.

This command should not pretend to know runtime state it cannot inspect.

### `config candidate`

Writes a timestamped candidate TOML fragment, never overwriting `~/.codex/config.toml`. It should include:

- selected profile block;
- rationale comments;
- known tradeoffs;
- manual merge instructions;
- CLI invocation equivalents when possible.

### `project inspect` and `project candidate`

These commands should support project-level uplift:

- inspect repo structure, existing `AGENTS.md`, `.codex/config.toml`, `.codex/hooks.json`, `.codex/rules/`, existing workflow docs, CI scripts, and verification commands;
- produce `.codex-uplift/observations.md` or similar;
- draft project-level candidates rather than overwriting files;
- propose per-project config only after noting that project `.codex/` layers load only when the project is trusted.

## 6. Component selection

v0.2 should support component-level install selection:

```text
--components home-agents,skills,agents,config,hooks,rules,plugin,project-skeleton
```

Default install should remain conservative:

```text
home-agents,skills,agents,config-fragment
```

Hooks, rules, plugins, project config, and high-autonomy profiles should be offered as candidates unless the user explicitly opts in.

## 7. Install modes

v0.2 should define three skill/plugin modes.

### Classic mode

Installs standalone skills into:

```text
~/.agents/skills/<skill-name>/SKILL.md
```

This is best for personal local use and low operational complexity.

### Plugin mode

Installs a plugin bundle and marketplace metadata without also installing standalone duplicate skills.

This is best when the user wants Codex-native plugin management or team distribution.

### Hybrid mode

Installs both standalone and plugin skills only when explicitly requested. It must detect duplicate skill names and print the expected ambiguity/precedence risk.

Hybrid should not be the default.

## 8. Autonomy profiles

v0.2 should introduce named posture profiles. These are not all recommended defaults; they are explicit modes with documented tradeoffs.

### `review-only`

Purpose: code review, exploration, research, or AGENTS.md drafting without edits.

Likely candidate:

```toml
[profiles.review-only]
approval_policy = "on-request"
approvals_reviewer = "user"
sandbox_mode = "read-only"
model_reasoning_effort = "high"
```

### `safe-interactive`

Purpose: normal interactive development with conservative technical boundaries.

Likely candidate:

```toml
[profiles.safe-interactive]
approval_policy = "on-request"
approvals_reviewer = "user"
sandbox_mode = "workspace-write"
allow_login_shell = false

[profiles.safe-interactive.sandbox_workspace_write]
network_access = false
```

### `autonomous-audited`

Purpose: the recommended high-autonomy profile for trusted local development while retaining sandbox boundaries and automatic review of eligible approval prompts.

Likely candidate:

```toml
[profiles.autonomous-audited]
approval_policy = "on-request"
approvals_reviewer = "auto_review"
sandbox_mode = "workspace-write"
allow_login_shell = false
default_permissions = "codex-uplift-autonomous"

[profiles.autonomous-audited.sandbox_workspace_write]
network_access = false

[permissions.codex-uplift-autonomous.filesystem.":project_roots"]
"." = "write"
"**/.env" = "none"
"**/.env.*" = "none"
"**/secrets/**" = "none"

[permissions.codex-uplift-autonomous.network]
enabled = false
```

This profile should pair with candidate rules/hooks and recovery gates:

- require clean or classified worktree before large tasks;
- create pre-task checkpoint branch or stash when appropriate;
- deny or prompt destructive git/file operations;
- record artifacts and verification;
- prefer small atomic commits when commits are requested;
- review delegated artifacts before integration.

### `trusted-power`

Purpose: trusted repos that need somewhat broader access but should not disable sandboxing.

Likely candidate:

```toml
[profiles.trusted-power]
approval_policy = "on-request"
approvals_reviewer = "auto_review"
sandbox_mode = "workspace-write"
allow_login_shell = false
default_permissions = "codex-uplift-trusted-power"

[profiles.trusted-power.sandbox_workspace_write]
network_access = false
# Add writable_roots only when the user knows the exact external paths needed.
```

This profile should prefer `--add-dir` or explicit `writable_roots` over `danger-full-access` when the real need is simply to write a second directory.

### `full-access-reviewed`

Purpose: the user's current high-autonomy default shape: no sandbox boundary, but eligible approval prompts route through automatic review.

Candidate:

```toml
[profiles.full-access-reviewed]
approval_policy = "on-request"
approvals_reviewer = "auto_review"
sandbox_mode = "danger-full-access"
allow_login_shell = false
```

This profile must carry a clear warning:

> This is not sandboxed safety. `auto_review` changes who reviews eligible approval prompts; it does not recreate the sandbox boundary. Use only in trusted repos or externally isolated environments, and pair it with rules/hooks, recovery checkpoints, and worktree hygiene.

For this profile, v0.2 should recommend candidate rules more strongly than hooks, because rules are specifically designed to control which commands Codex can run outside the sandbox.

### `external-isolated`

Purpose: devcontainer/VM/container environment where the external environment, not Codex's own sandbox, is the main isolation boundary.

Candidate:

```toml
[profiles.external-isolated]
approval_policy = "on-request"
approvals_reviewer = "auto_review"
sandbox_mode = "danger-full-access"
allow_login_shell = false
```

This profile should require an explicit environment note:

- what is the outer boundary;
- what secrets are mounted;
- whether network is constrained;
- whether rollback is available;
- whether Codex credentials are exposed inside the environment.

### `ci-noninteractive`

Purpose: automation where prompts cannot be handled.

Candidate:

```toml
[profiles.ci-noninteractive]
approval_policy = "never"
sandbox_mode = "workspace-write"
allow_login_shell = false

[profiles.ci-noninteractive.sandbox_workspace_write]
network_access = false
```

This must not be suggested for normal local interactive work.

## 9. Sandbox/permission model guidance

v0.2 documentation should say:

- The sandbox is a technical boundary for command execution.
- Approval policy decides when Codex must pause before crossing relevant boundaries or executing actions that require approval.
- `auto_review` changes the reviewer for eligible approval prompts; it does not itself add a sandbox or cause every risky action to be reviewed.
- `danger-full-access` should be treated as a full-access execution posture, even if `approval_policy = "on-request"` and `approvals_reviewer = "auto_review"` still apply to some prompt categories.
- Rules are the better mechanism for command-prefix allow/prompt/forbid behavior outside the sandbox.
- Hooks can be useful lifecycle guardrails, but they are not complete enforcement boundaries.

The package should avoid making universal claims like “danger-full-access lets many risky actions run without review” unless it has local probe evidence. The auditable statement is narrower:

> When `sandbox_mode = "danger-full-access"`, the sandbox boundary is disabled for command execution. Therefore, approval prompts that would have been caused specifically by sandbox escalation may not occur. Other approval paths may still occur depending on `approval_policy`, granular policy, rules, apps, hooks, skills, and tool behavior.

## 10. Rules and hooks strategy

v0.2 should add candidate rules files, not just hooks.

Suggested candidate rules:

```python
prefix_rule(
    pattern = ["rm", "-rf", "/"],
    decision = "forbidden",
    justification = "Never remove filesystem root."
)

prefix_rule(
    pattern = ["git", "push", "--force"],
    decision = "prompt",
    justification = "Force push rewrites remote history; require review."
)

prefix_rule(
    pattern = ["git", "reset", "--hard"],
    decision = "prompt",
    justification = "Hard reset can discard work; require review and checkpoint."
)

prefix_rule(
    pattern = ["git", "clean"],
    decision = "prompt",
    justification = "git clean can delete untracked files; require review."
)
```

Hooks should remain sample-only by default, with candidate enablement paths:

- `PreToolUse`: block obviously destructive patterns and require evidence of checkpoint for certain commands.
- `PermissionRequest`: log and classify approval requests, especially under `auto_review`.
- `Stop`: require final response audit shape for substantive tasks when explicitly enabled.

## 11. Recovery and checkpointing

High-autonomy profiles need compensating controls. v0.2 should add an optional recovery workflow:

```text
codex-uplift-init checkpoint create
codex-uplift-init checkpoint status
codex-uplift-init checkpoint recover
```

At minimum, the project-init skill and home `AGENTS.md` should instruct orchestrators to:

- inspect dirty worktree before edits;
- classify existing changes as user/team/agent-generated when possible;
- create a branch, stash, or patch checkpoint before large autonomous edits;
- commit in small slices when commits are requested;
- preserve audit artifacts;
- record verification and boundaries.

This does not eliminate risk, but it makes risk recoverable.

## 12. Project-level uplift

v0.2 should extend beyond user-level installation into project-tailored candidates:

```text
codex-uplift-init project inspect
codex-uplift-init project candidate --include agents-md,config,rules,hooks,skills
```

The project initializer should produce:

- `AGENTS.md.candidate` or a patch for `AGENTS.md`;
- `.codex/config.toml.candidate` for repo-scoped config;
- `.codex/rules/*.rules.candidate` for command rules;
- `.codex/hooks.json.candidate` and hook scripts if hooks are appropriate;
- `.codex-uplift/observations.md` containing observed facts, inferred conventions, and open questions;
- `.codex-uplift/setup-plan.md` explaining what should be applied and why.

Project config should be conservative because project `.codex/` layers are trusted-gated. The setup assistant should surface that trust boundary instead of silently depending on project-local config.

## 13. App vs CLI support

v0.2 should explicitly support both Codex app and CLI, but not assume identical controls.

Documentation should provide:

- app-oriented guidance for using settings, plugins UI, auto-review, and custom permissions where available;
- CLI-oriented guidance for `--profile`, `--config`, `--sandbox`, `--ask-for-approval`, `--add-dir`, `--full-auto`, and `--yolo`;
- warnings where app modes and CLI flags do not map one-to-one;
- a `doctor` output field that says whether a setting was observed from config, inferred from CLI flags, or not observable.

## 14. Installer tests required for v0.2

v0.2 should not ship without tests for:

- fresh install into temp user/Codex homes;
- existing files produce `.candidate.*` files;
- `--force` creates backups before overwriting;
- component selection installs only selected components;
- classic/plugin/hybrid modes behave as documented;
- duplicate skill names are detected;
- plugin marketplace path resolves correctly;
- config candidates parse as TOML;
- JSON templates parse;
- rules examples pass `codex execpolicy check` when Codex is available;
- hook scripts emit valid output shapes;
- `.DS_Store` and other system files are ignored;
- uninstall removes only files listed in the install manifest.

## 16. Bottom line

The revised response should say:

> The reviewer did not invent a new product category; it made explicit the setup-assistant ambition already implied by the user's goals. v0.1 was a reasonable bootstrap slice, but v0.2 should now deliberately become the first setup-assistant version: inspect current state, generate non-destructive candidates, support component selection, define autonomy profiles, handle project-level tailoring, and pair maximal autonomy with explicit compensating controls.

> More autonomy should require more explicit recovery, auditability, and posture inspection — not merely fewer prompts.
