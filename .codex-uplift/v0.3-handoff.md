# v0.3 Handoff

Date: 2026-04-30

## v0.3 Candidate Capabilities

- Rich project posture determiner with evidence/confidence.
- Exact or probed config behavior across CLI/app/IDE.
- Config merge assistant for active user/project config.
- Fuller project candidate generation for `AGENTS.md`, `.codex/config.toml`, hooks, and rules.
- Compaction prompt registry, fixtures, and automated/manual evaluation outputs.
- Tool-output-filter evaluation harness, including RTK only if evaluation gates pass.
- Pre-session posture router with `PostureRecommendation` artifacts.
- Plugin lifecycle/update checks beyond local install metadata.
- Team/org policy-pack seams.

## v0.2 Seams Preserved

- Stable install component IDs.
- Explicit install modes.
- Machine-readable manifest entries with target paths and hashes.
- Observed/inferred/proposed/unknown config doctor categories.
- Candidate-only config/profile generation.
- Candidate-only project/hooks/rules/compact/RTK commands.
- RTK kept as evaluation-only, not a dependency.
- Hook/rule/full-access surfaces remain inactive by default.

## Unresolved Empirical Questions

- Does live Codex accept absolute local marketplace `source.path` values for personal marketplace entries outside the user-home root?
- How does each Codex client surface plugin cache/restart behavior after local marketplace updates?
- What exact profile keys are accepted in the installed local Codex version?
- How should generated project candidates vary by repo maturity without over-classifying?
- Can default Codex compaction behavior be observed safely enough for candidate prompt comparison?

## Recommended First v0.3 Slice

Implement `project candidate` as a real two-stage determiner:

1. `project inspect` writes structured observations with evidence and unknowns.
2. `project candidate` writes project `AGENTS.md`, `.codex/config.toml`, hook, and rule candidates from those observations.
3. Tests cover candidate-first behavior, trust caveats, and no active `.codex` mutation without explicit approval.

## Release Feedback To Gather Before v0.3

- Does `install --mode plugin` appear correctly after Codex restart?
- Do users understand `classic`, `plugin`, and `hybrid`?
- Are config profile candidates useful or too verbose?
- Does manifest/status/uninstall match user expectations?
- Which candidate-only seams do users reach for first?

## Should v0.3 Wait?

Begin v0.3 only after the user decides whether to ship, ship-alpha, hold, or revise the v0.2 release candidate.
