# v0.2 Implementation Plan

Date: 2026-04-30

## Orchestration Model

The orchestrator owns scope, specs, deferrals, integration review, validation, and commits. Implementation details should be delegated in bounded slices with disjoint write scopes. Delegates must not mutate real user-home config during tests.

Exploration delegates should normally run at medium reasoning. Higher reasoning should be reserved for tasks with genuine ambiguity or high-stakes design coupling. The orchestrator should make medium-reasoning exploration successful by writing prompts that include:

- exact files/directories to inspect;
- known pitfalls to verify rather than assume;
- official-doc or line-reference evidence requirements;
- clear “do not edit” boundaries for explorers;
- expected return schema;
- explicit unknowns to preserve instead of guessing;
- acceptance tests or review checks the explorer should propose.

## Proposed Command Surface

Minimum v0.2 command seams:

```text
codex-uplift-init inspect
codex-uplift-init install --mode classic|plugin|hybrid|minimal
codex-uplift-init status
codex-uplift-init uninstall --manifest <path>
codex-uplift-init config doctor
codex-uplift-init config candidate --profile <profile>
codex-uplift-init project inspect
codex-uplift-init project candidate --include <surfaces>
codex-uplift-init rules candidate --profile <profile>
codex-uplift-init hooks candidate --profile <profile>
codex-uplift-init compact candidate --profile <profile>
codex-uplift-init rtk evaluate --plan-only
codex-uplift-init verify
```

Commands that cannot be fully implemented in the first slice must return a useful candidate/report artifact and point to an explicit deferral ID.

## Install Modes

- `classic`: home agents, standalone skills, agents, config candidates, hook samples, rule samples, manifest.
- `plugin`: home agents, agents, config candidates, hook samples, rule samples, plugin, plugin marketplace, manifest; standalone skills skipped by default.
- `hybrid`: classic plus plugin; must print duplicate skill warnings and require explicit duplicate acknowledgement if implemented.
- `minimal`: home agents candidate, config candidates, manifest only.

## Config Doctor And Candidate Behavior

- `config doctor` reads config presence and selected keys when parseable.
- It reports `observed`, `inferred`, `proposed`, `unknown`, and `requires_probe`.
- It does not claim exact effective behavior across app/CLI/IDE or managed layers.
- `config candidate --profile` writes a candidate TOML file and companion notes.

Profiles to support:

- `review-only`
- `safe-interactive`
- `autonomous-audited`
- `install-window`
- `net-limited`
- `full-access-reviewed`
- `external-isolated`
- `ci-noninteractive`

## Project Setup Candidate Behavior

- `project inspect` writes `.codex-uplift/project/observations.md`.
- `project candidate` writes candidate project files only:
  - `AGENTS.md.candidate.<timestamp>`
  - `.codex/config.toml.candidate.<timestamp>`
  - `.codex/hooks.json.candidate.<timestamp>`
  - `.codex/rules/*.candidate`
- It records trust caveats and does not assume project `.codex/` config is active.

## Rules And Hooks Candidate Behavior

- Hooks remain samples/candidates by default.
- Rules remain samples/candidates by default.
- Candidate files include policy IDs and warnings.
- Generated docs state hooks are lifecycle guardrails, not complete enforcement; rules control commands outside the sandbox and are experimental.

## Manifest, Status, And Uninstall Behavior

Manifest should live under the active Codex home unless overridden:

```text
<codexHome>/codex-uplift-kit/manifest.json
```

Each entry should include:

- package name/version;
- component ID;
- kind;
- source path;
- target path;
- action type;
- sha256 when content was written;
- install mode;
- timestamp.

`status` should classify entries as present, missing, modified, candidate, or unknown.

`uninstall` should remove only package-owned files whose current sha256 matches the manifest. Modified files should be left in place and reported.

## Test Plan

Delegate tests should cover:

- `npm run smoke`
- `npm test`
- `npm pack --dry-run`
- temp-home fresh install;
- temp-home existing-file candidate behavior;
- force backup behavior;
- plugin mode skips standalone skills;
- hybrid mode reports duplicate skill names;
- generated marketplace `source.path` resolves to copied plugin;
- manifest status reports installed files;
- uninstall leaves modified files;
- config candidates are written, not merged;
- hook output shape tests still pass.

## Rollback Plan

- Keep all changes in local commits.
- Use manifest-aware uninstall for installed package-owned files in temp homes.
- Do not mutate real user-home config.
- If implementation delegates overreach, revise or reject their patch rather than trying to normalize unsafe behavior.

## Delegation Work Packages

Pending delegate inspection, likely implementation packages are:

1. CLI parser and command-dispatch skeleton.
2. Install mode/component registry plus plugin path and duplicate skill fix.
3. Manifest/status/uninstall.
4. Config/profile candidate generator.
5. Project/hooks/rules/compaction/RTK candidate surfaces.
6. Tests and docs.

## Delegation Prompt Requirements

Each implementation worker must receive:

- ownership scope and non-overlap with other workers;
- explicit note that other agents may also be working and they must not revert others' edits;
- target behavior and safety boundaries;
- tests they own adding or updating;
- paths they may edit;
- paths they must not edit;
- final return shape with files changed, verification run, and unresolved risks.
